# One Piece RPG

Single-player, local-first narrative RPG set in the One Piece world in the post-Egghead era.
You create a 17-year-old character, set sail from the East Blue on a small boat and navigate an
open world of 332 islands with combat, Akuma no Mi, Haki, bounty, nemesis, crew, alliances,
poneglyphs and Laugh Tale. The game never ends: every ending you reach becomes an epilogue and
the campaign carries on.

All inference runs on the **player's own Claude Max subscription**, via
[CLIProxyAPI](https://github.com/router-for-me/CLIProxyAPI), with no paid API key and no
local model. The story is 100%
generated by the models; the engine only handles bookkeeping (state, persistence,
validation) and narrative authorship belongs entirely to the models.

> Player guide (character creation, acting, META, Den Den Mushi, endings):
> **[`docs/PLAYER_GUIDE.md`](docs/PLAYER_GUIDE.md)** (PT-BR: [`docs/PLAYER_GUIDE.pt-BR.md`](docs/PLAYER_GUIDE.pt-BR.md)).

## Installation (release)

The game ships as a desktop app on the Releases page, with its own window. The package
bundles the backend, the frontend and the CLIProxyAPI binary; the player's machine needs no
Python, Node or browser.

1. Download the latest release.
2. Open the app.
3. On first run, the setup wizard authenticates your Claude Max account via OAuth.
4. Create your campaign and character. Progress lives in a local SQLite database and
   preferences in an `app_settings.json`, both in the app's data folder.

Running from source remains supported; see the
[Development](#development-from-source) section.

## Architecture

Multi-agent pipeline with fixed roles. Every role runs on `claude-sonnet-5` by default
(configurable per role via env). The Narrator writes all the prose, including NPC dialogue.
Every other agent emits structured JSON only, via tool use.

| Role | Responsibility |
|---|---|
| Director PRE | Composes the scene (location, cast, tension), moves NPCs, decides transitions and navigation |
| NPC Minds | Subjective snapshot of each NPC in scene (emotion, goal, memory) |
| Narrator | Authors the entire scene in prose (social and combat), emits the structured `turn_meta` |
| Director POST | Applies world deltas (alignment/chaos/bounty/reputation/belly), dispatches generators |
| Auditor | Post-turn gate: validates prose, presence and cards before revealing to the player |
| Crystallizer | Extracts long-term factual memory when the scene closes |
| Meta Router | Classifies META directives (question/remember/forget) outside the turn flow |

Models and temperatures are configurable via env (`NARRATOR_MODEL`, `AGENT_MODEL`,
`DIRECTOR_MODEL`, `CRYSTALLIZER_MODEL`, `META_ROUTER_MODEL`, `AUDITOR_MODEL`,
`AGENT_TEMPERATURE`, `META_ROUTER_TEMPERATURE`, etc.; see `backend/.env.example` and
`backend/app/config.py`).

### Anatomy of a turn (DO action)

1. **Director PRE** reads the player's action, recent prose and world state. It emits the
   scene (`location`, `ambient`, `tension_level`, `area_slug`), the present cast
   (`npcs_in_scene`), NPC movement (`npc_location_updates`, the single enter/exit channel),
   scene transition (`gancho_de_lugar` or `elipse_de_tempo`), navigation and tactical
   signals (`surprise_actions`).
2. **Reconcile-on-return**: a frozen NPC who reappears in scene goes through a
   reconciliation call that updates situation, goal, mood and world awareness, coherent
   with elapsed time and crystallized memory. NPCs never leave your story, come back and
   forget what they were doing, no matter how many turns have passed.
3. **NPC Minds** run in parallel for the cast in scene and produce `mind_snapshots` with no
   pre-scripted decision. The Narrator decides each one's tactics, speech and gestures.
4. **Narrator** receives the turn state (scene, player sheet, snapshots, crystals, recent
   prose, island context) and emits through the `emit_turn` tool: the reflexive
   `pre_emit_audit` gates, the prose (streamed over the WebSocket) and the structured
   `turn_meta` (NPCs/items/ships to generate, tactical outcomes, recruitment resolutions,
   crew offers, scene status, imagery used).
5. **npc_mind_post**: after the prose, each NPC in scene updates their mind (emotion,
   relationship deltas, memory note, goal progress).
6. **Director POST** applies the qualitative deltas to the world, records events (combat,
   ship, comms, world events), updates nemesis and endgame and dispatches
   `dispatched_jobs` to the content generators.
7. **Auditor** runs before the prose is revealed. It cross-checks prose, full cards of the
   NPCs in scene, player sheet, deltas and memory. It can rewrite the prose, fix cards,
   adjust presence, mint an NPC the prose staged without a card (`mint_npc`) and flag
   duplicates for merge. Best-effort: on timeout or error, the original prose passes intact.
8. **Scene close**: with `scene_status = fecha` (nudged at `SCENE_TURN_CAP = 8`, forced at
   `SCENE_HARD_CAP = 16`) the Crystallizer extracts crystals, and each NPC leaving the
   scene is frozen with a `departure_snapshot` (freeze-on-close).

### META flow

META inputs go through the `meta_router` without advancing the turn: `question` answers out
of character with access to game state (including the player sheet), `remember` writes
permanent directives to the database, `forget` lists the active directives for deactivation
in the UI.

## Inference layer

Core file: `backend/app/proxy/client.py`.

- **Cloaking**: CLIProxyAPI in OAuth mode drops the `system` field, so system prompts
  travel inside the first user message, wrapped in per-role XML tags
  (`<narrator-instructions>`, `<director-instructions>`, ...).
- **Prompt caching**: `build_content()` assembles the message in blocks ordered from static
  to volatile: instructions + tools with `cache_control {type: ephemeral, ttl: "1h"}`
  (header `anthropic-beta: extended-cache-ttl-2025-04-11`), semi-static `cached_sections`
  (cast catalogs, SVG map, crystals) with their own breakpoint, and finally the dynamic
  sections at full price. Cached sections render byte-stable across turns to preserve the
  hit. A turn drops from ~134k to ~34k billed tokens from the scene's 2nd turn onward.
- **Three primitives**: `call_tool` (forces a specific tool and returns the parsed input),
  `call_text` (plain text) and `stream_text` (prose streaming over the WebSocket).
- **Honest error runtime**: exhausted subscription quota becomes `QuotaExceededError`
  (HTTP 503 + `Retry-After`); the turn pauses with no partial persistence and resumes when
  quota returns. A model safety refusal becomes `ModelRefusalError` (HTTP 422); the UI
  returns the action to the composer for rephrasing and the campaign stays intact.
  Transient burst 429s are retried by the SDK with backoff. The engine respects the
  model's refusals.
- **Tracing**: every call records tag, model, input sections, output and real `usage`
  (`input`, `output`, `cache_read`, `cache_creation`) in a per-turn buffer, displayed in
  the frontend devtools panel.

## Prose quality

Prose vices are closed through the schema, with textual rules as support only:

- **Reflexive `pre_emit_audit` gates**: `required` string-enum fields in the schema of the
  prose-writing tools (Narrator, NPC minds, Auditor, side-prose generators). Each gate
  names a vice (recap, contrastive framing, gesture glossing, aphorism, fragmentation,
  player agency, template scenery) and forces the model to re-read the commitment before
  emitting. The engine discards the field at parse time; the gain comes from the act of
  re-reading.
- **Anti-autophagy ledger**: the Crystallizer maintains `overused_imagery` (8-turn window,
  cap 10) with recent imagery, fed back to the Narrator as a "vary from these" list.
- **Auditor scoped by prose kind**: `turn` (full audit), `opening`, `timeskip_recap` and
  `epilogue` (form and agency, no mechanical accounting, compression-by-design in recaps).

## Memory and continuity

- **Crystals**: facts extracted at scene close, with category (relationship, event,
  revelation, promise, combat outcome, world fact, ...), participants, witnesses and
  hidden witnesses (the basis of the Auditor's anti-omniscience). Updatable and
  obsoletable; FTS5 search, no embeddings.
- **Per-NPC mind** (`agent_state.py`): personal event log (window 20 + important cap 30),
  relationships with affinity clamped to [-1, 1] and `bond_tier` 0/1/2, mood and goal
  progress. Spatial salience decides what each NPC perceives of what happens far from them.
- **Evolving world by freezing**: an NPC leaving the scene is frozen with an executive
  snapshot (where they stayed, what they pursued, the player's last directive). On return,
  reconciliation redraws the volatile state anchored in crystals and elapsed time.
  Identity (name, tier, origin, core bond) is immutable.
- **Foreshadow pool**: continuity hooks planted by the Director/Auditor, with theme, origin
  island and age in turns, offered to the Narrator for future payoff.

## Content generators

Triggered by an AND of two signals: the Narrator marks the need in `turn_meta` and the
Director POST confirms via `dispatched_jobs`. All run with model-side dedup: a generator
can answer `duplicate_of_existing_id` and the engine reuses the existing card instead of
minting a duplicate.

| Generator | Produces |
|---|---|
| `npc_generator` | Wiki-style card (sex, appearance, history, personality, expressiveness, tier, Haki, fruit, Marine moral code) + agent mind; style reference comes from 10 real wiki characters |
| `creature_generator` | Card-only creature (pet, mount, Sea King), no mind, rendered via `behavior_notes` |
| `item_generator` | ITEM card; when `acquired_by_player`, it enters the inventory with quantity and origin note |
| `ship_generator` | SHIP card; when acquired, performs the full fleet swap (active/reserve, Jolly Roger migrates) |
| `island_designer` | Name and physical/cultural context for an invented island, cached per campaign |
| `research` | Canonical island briefing via the Fandom MediaWiki API (query planner, executor, synthesizer), cached per island; degrades without blocking if the API fails |
| `fruit_alt_canon` | One hook per campaign when the player picks a fruit with a canon owner: the generator judges whether the NPC being born is the displaced owner |

## World and navigation

- **Full catalog**: 332 entries in `world_islands.json` (301 navigable islands + 31
  non-navigable containers such as the Red Line and groupings), with coordinates, cluster,
  region and canon flag.
- **Semantic SVG map**: `world_map_svg()` generates a stable SVG (one `<circle>` per
  island, groups per sea, the fixed Log Pose route in Paradise, the Red Line as reference
  geography) injected into the Director's cached block. The Director navigates by reading
  the map, with no deterministic suggestion list.
- **Fog of war**: every island has `discovered`/`visited`/`discovered_via` (canon-known,
  NPC mention, News Coo, visited). Revelation is monotonic.
- **Crossings**: travel days from calibrated euclidean distance (`_UNITS_PER_TRAVEL_DAY`),
  multiplied by the ship's `speed_class` (raft 0.7 up to exceptional 2.0) and hull
  penalty. `set_sea` opens the crossing (destination chosen by the Director, including
  when the player asks by criterion, such as "the nearest island"), `set_adrift` covers
  aimless sailing, and the at-sea position is interpolated for the map pin.
- **Invented islands**: allocated to real land slots (`blank_island_slots.json`),
  restricted to clusters that admit generation. The East Blue is canon-only.
- **Clock and timeskip**: `campaign_day` and derived ages; a timeskip short-circuits the
  turn with a batch executor (retroactive per-NPC logs + world events), clock advance,
  clamped tier-up and a cinematic recap.
- **News Coo**: the Director organically decides when the newspaper arrives, fed by a pool
  of signals (bounty jumps, unpublished world events); the Narrator stages the delivery
  and the edition is archived in the Newspaper tab.
- **Den Den Mushi and Vivre Cards**: mushis paired per NPC (baby with island range,
  standard and visual global), black mushi for wiretapping, white mushi for
  counter-surveillance, Buster Call flag. Vivre cards with visual state
  (white/burning/errant/ashes) and 8-point compass direction to the owner.

## Game systems

- **Character creation**: deterministic trait roll (1d4+1, mutual exclusions), validation
  against YAML catalogs of classes and fruits, age locked at 17, starting tier capped at
  STRONG. Everything is editable afterwards.
- **Combat**: the Narrator authors combat scenes like any other. Tactical outcomes
  (surrender, hostage capture) arrive through `turn_meta` and the engine applies them with
  mechanical guards (dead/missing/already captured cannot be captured).
- **Techniques**: registry with case-insensitive upsert per owner, use count and
  description backfill; persists for player, crew and nemesis.
- **Fighting style**: consolidated from scratch at every tier-up (fruit + Haki +
  techniques + traits + recent combat), stored as summary and tags.
- **Breakthroughs**: six kinds (fruit awakening, black blade, Haoshoku infusion, Voice of
  All Things, advanced Armament, advanced Observation), confirmed by the Director and
  described in dedicated prose, mirrored into the affected cards.
- **Crew**: membership lives on the NPC card (`player_crew`/`ex_player_crew`), recruitment
  resolved narratively with engine guards (an ex-member awaits reconciliation),
  dissatisfaction accumulates with decay, soft cap of 10 signaling a fleet.
- **Alliances and factions**: pacts between crews with hierarchy (peer, subordinate,
  sovereign) and anti-invention (the other crew must exist); per-faction reputation in
  [-2, +2] with ally/neutral/hostile buckets, the captain's 3x weight on crew reputation
  and a multiplier effect on recruitment.
- **Marine Nemesis**: a singular, evolving antagonist (ladder from Captain to Fleet
  Admiral), with evolution milestones, posture, permanent death with a successor gap and a
  parallel nemesis (a promoted bounty hunter).
- **Economy**: belly with qualitative per-tier deltas (small up to absurd) sampled by the
  engine, event-based inventory (acquired/lost/consumed/given) with stacking.
- **Fleet**: invariant of exactly one active ship, hull condition
  (pristine/scarred/damaged/broken) affecting speed, Jolly Roger declarable from the UI.
- **Poneglyphs and endgame**: poneglyphs as ITEM cards (road/rio/historical), one-time
  content revelation once transcribed and translated; a qualitative consummated-state
  detector (`emit_endgame_state`) accumulates world flags (Imu, Mary Geoise, territories),
  Laugh Tale revelation (4 Road, or 3 + Voice) and reached endings (Pirate King, Yonkou,
  Fleet Admiral, revolutionary leader, ...). Each ending generates a ~1200-1500 word
  epilogue and the game continues.

## In-game editing

All state is editable from the HUD at no turn cost and with no re-crystallization: whole
NPC cards (appearance, personality, history, fruit, Haki, alignment, flags), player sheet,
techniques, breakthroughs, fruit usage log, memory crystals, any turn's prose. The merge is
defensive by whitelist (`edit.py`); unknown fields are ignored and the next turn reads the
new state. Beyond editing: **rewind** undoes the last turn via the pre-turn world snapshot
(`world_snapshots`) and returns the action text to the composer; **reroll** re-runs only
the Narrator over the same input, with an optional one-shot instruction.

## HTTP API

Main groups (routers in `backend/app/api/`):

| Group | Routes |
|---|---|
| Health and setup | `GET /api/health`, `GET /api/player-guide`, `GET/PUT /api/settings`, `POST /api/setup/connect-claude` (proxy OAuth) |
| Catalogs | `GET /api/catalog[/traits|/classes|/fruits]`, `POST /api/catalog/roll-traits` |
| Campaigns | `POST/GET/DELETE /api/campaigns[/{id}]`, `POST /api/campaigns/{id}/character` |
| Turn | `POST /api/campaigns/{id}/turn` and WebSocket `/api/campaigns/{id}/turn` (streaming of prose, deltas and trace) |
| State | `GET .../world`, `/news`, `/comms`, `/economy`, `/fleet`, `/factions`, `/alliances`, `/crew`, `/ending`, `/poneglyphs`, `/cards`, `/search` (FTS5), `/directives`, `/techniques` |
| System commands | `POST .../turns/{idx}/rewind`, `POST .../turns/{idx}/reroll-prose`, `PATCH .../turns/{idx}/prose` |
| Editing | `PATCH/DELETE` on `/cards/{id}`, `/player`, `/techniques/{id}`, `/breakthroughs/{kind}`, `/fruit-usage/{idx}`, `/crystals/{id}`; `POST .../jolly-roger`, `POST .../crew/offers/{npc_id}` |

Inference errors map to `503 quota_exceeded` (with `Retry-After`) and
`422 model_refusal`.

## Persistence

A single SQLite per install (`backend/data/campaign.db`), WAL, foreign keys on,
`busy_timeout` 30s. Eleven pure-SQL migrations applied on boot in numeric order against
`PRAGMA user_version`. Core tables: `campaigns`, `turns` (append-only, with per-turn
`trace`), `world_snapshots` (pre-turn state that powers rewind), `story_cards` (free-form
schema in `data_json` for player, NPCs, items, ships and factions), `crystals` (+ FTS5 with
sync triggers), `game_clock`, `directives` and creation catalogs. App preferences (theme,
language, proxy, auto-spawn) live in `data/app_settings.json`, outside the campaign
database.

## Frontend

Svelte 5 with runes, no UI lib, its own design system in `app.css` (dark-first, light theme
via `data-theme`, bundled Inter Variable). Screens: title with proxy/database status and
campaign management, setup wizard with Claude OAuth (health polling until authentication
shows up), character creation with trait rolling, and the game screen. In game: streaming
prose over WebSocket (reconnection with exponential backoff), DO/META composer, tabbed HUD
(sheet, crew, fleet, factions, alliances, economy/inventory, newspaper, comms, memory with
FTS search, techniques, poneglyphs/endings), Leaflet map with bundled tiles, at-sea
interpolated position pin and vivre card overlay, inline editor for everything the editing
API exposes, per-turn rewind/reroll, toasts (newspaper, mushi, crew offer, ending reached)
and a devtools panel with the full trace of every LLM call (model, input sections, output,
full-price and cached tokens). The UI is bilingual (pt-BR/EN) with live switching.

## Prompts

75 files in `prompts/*.pt-br.md`, the single source of behavior: masters for the Narrator,
Director, agent, Auditor and meta_router, plus conditional addenda per subsystem (combat,
navigation, mushi, recruitment, nemesis, timeskip, News Coo, ...) and generator prompts.
They are read from disk on every call: editing a prompt takes effect on the next turn, no
restart. Prompts are written in PT-BR; the prose comes out in the campaign's language
(PT-BR or EN).

## Development (from source)

Prerequisites: [uv](https://docs.astral.sh/uv/), Node 18+ (tested on 22) and an active
Claude Max subscription. Download the CLIProxyAPI binary from the
[upstream releases](https://github.com/router-for-me/CLIProxyAPI/releases) into
`cliproxyapi/bin/` and create `cliproxyapi/config.yaml` from
`cliproxyapi/config.example.yaml`. The backend tries to spawn the binary on boot
(auto-spawn) when it is present and the option is enabled; OAuth authentication is done
through the UI's own wizard. Alternatively, run the proxy standalone.

### Two processes (frontend hot-reload)

Backend (terminal 1), from inside `backend/`:

```bash
cp .env.example .env        # adjust if the proxy runs on another port
uv sync
uv run uvicorn app.main:app --reload --port 8400
```

Frontend (terminal 2), from inside `frontend/`:

```bash
npm install
npm run dev                 # Vite on :5173, proxying /api to :8400
```

Open **http://localhost:5173**.

### Single process (local web)

```bash
cd frontend && npm run build      # generates frontend/dist/
cd ../backend && uv run uvicorn app.main:app --port 8400
```

Open **http://localhost:8400**: FastAPI serves the Svelte build and the API on the same
origin.

## Configuration

Main variables (defaults in `backend/app/config.py`, overrides in `backend/.env`):

| Variable | Default | Purpose |
|---|---|---|
| `ANTHROPIC_PROXY_URL` / `ANTHROPIC_PROXY_KEY` | `http://127.0.0.1:8318` / `onepiece-proxy-key` | CLIProxyAPI address and key |
| `NARRATOR_MODEL` | `claude-sonnet-5` | Prose (Narrator and Auditor) |
| `AGENT_MODEL`, `DIRECTOR_MODEL`, `CRYSTALLIZER_MODEL`, `META_ROUTER_MODEL` | `claude-sonnet-5` | Structured decisions |
| `AGENT_TEMPERATURE` | `0.7` | Agent/Director temperature (see note) |
| `OPRPG_DB_PATH`, `OPRPG_PROMPTS_DIR`, `OPRPG_FRONTEND_DIST` | empty | Path overrides |

> **Full Sonnet 5.** Every role runs on `claude-sonnet-5` with adaptive thinking (default,
> on). Sonnet 5 **rejects** non-default `temperature`/`top_p`/`top_k` (400 error):
> `proxy/client.py` (`_accepts_temperature`) does not send `temperature` to that model, so
> `AGENT_TEMPERATURE` stays inert while the model is Sonnet 5 (kept only for reverting to
> an older model). The forced `tool_choice` of the structured tools coexists with adaptive
> thinking. Sonnet 5's tokenizer produces ~30% more tokens than Sonnet 4.6, so the per-turn
> cache budget is larger — measure real `usage` in the devtools panel.

## Operations

Uvicorn runs without `--reload` in production; Python code changes require a manual
restart. Prompt/addendum changes need no restart (read from disk per call).

## Open source distribution

Each user runs the game on their own Claude Max subscription; the project needs no shared
key and no hosted backend. The official release ships the desktop app, and the source stays
open for anyone who prefers to build. Data credits and references in
[`CREDITS.md`](CREDITS.md).

> Packaging must include, beyond the app's window shell: the bundled backend, the frontend
> build (`frontend/dist/`), the `prompts/` folder (read from disk at runtime), the world
> data and catalogs from `backend/app/db/`, the SQL migrations and the CLIProxyAPI binary
> with its `config.yaml`.
